# constants.py â€” Spec

## Purpose
Holds palette (light/dark theme dicts), the candlestick bull/bear colors, the intervalâ†’yfinance-period lookup, and a handful of small helpers for intraday session classification and interval arithmetic. Acts as the single source of truth for display colors and interval-related constants, avoiding circular imports.

## Public API
- `BULL_COLOR` / `BEAR_COLOR` â€” candlestick body/wick colors. Default teal-green `#26a69a` / coral-red `#ef5350`; resolved once at import from `settings.get("use_colorblind_palette")` (Okabe-Ito orange `#e69f00` / sky-blue `#56b4e9` when on) and **mutated in place** at runtime by `ChartApp.set_use_colorblind_palette`. Every directional-sentiment color must derive from these (see the directional-palette helpers below). Audit `color-blind-palette-audit`.
- **Directional-sentiment palette helpers** (single chokepoint so every bull/bear/up/down/gain/loss/MFE-MAE color follows the Okabe-Ito toggle):
  - `colorblind_palette_active() -> bool` â€” `BULL_COLOR == _COLORBLIND_BULL_COLOR` (live, reflects the in-session toggle).
  - `recolor_to_hue(base_hex, hue_source_hex) -> str` â€” recolour `base_hex` to `hue_source_hex`'s hue, **preserving** `base_hex`'s lightness + saturation (HLS). Lets a tuned green/red *tint* swing onto the Okabe-Ito hue while keeping its tone.
  - `sentiment_recolor(base_hex, *, bullish) -> str` â€” pass-through under the default palette (pixel-exact); under Okabe-Ito recolours `base_hex` to the live `BULL_COLOR`/`BEAR_COLOR` hue. The function every directional color routes through.
  - `bull_row_bg(theme)` / `bear_row_bg(theme)` â€” watchlist/table row-tint accessors (`sentiment_recolor(theme["bull_row_bg"], â€¦)`).
  - `macd_histogram_palette() -> (strong_bull, weak_bull, weak_bear, strong_bear)` â€” live Okabe-Ito-aware 4-class histogram palette built from `BULL_COLOR`/`BEAR_COLOR` + the `_BULL_TINT_PALE`/`_BEAR_TINT_PALE` (`#b2dfdb`/`#ffcdd2`) module tints. `is_app_macd_palette(palette)` recognises the default + Okabe variants so the renderer can swap them for the live palette while still honouring a genuinely custom 4-tuple.
  - Status colors (error/warn/info/ok in `gui/colors.py`) are a **different axis** and deliberately do NOT route through these.
- `LIGHT_THEME: dict`, `DARK_THEME: dict` â€” palettes keyed by named slot (`win_bg`, `fig_bg`, `ax_bg`, `text`, `text_disabled`, `grid`, `spine`, `input_border`, `tree_bg`, `tree_fg`, `bull_row_bg`/`fg`, `bear_row_bg`/`fg`, `tooltip_bg`/`fg`, `watermark`, `pre_shade`, `post_shade`, `crosshair`). The `text_disabled` slot is the foreground for disabled menu entries (e.g. empty recent-file placeholders); GitHub-muted greys (`#8b949e` light, `#6e7681` dark) avoid the Windows-default etched/embossed disabled-text style that looks blurry against dark backgrounds. The `input_border` slot is **distinct** from `spine` â€” `spine` is for chart axis lines (often subtle), but `input_border` MUST visibly contrast against `ax_bg` so users can see clickable input tap targets (`TEntry`/`TCombobox`/`TSpinbox`). Audit `input-border-visible`: commit `536fe6c` previously collapsed input borders to the `spine` color, making them blend into the field background; the test at `tests/unit/gui/test_input_border_visibility.py` pins minimum RGB contrast and forces the input widgets to use `input_border`. `gui.menu_theme` consumes `win_bg`/`text`/`grid` to force classic `tk.Menu` widgets onto the dark palette and appends a Tk-rendered cascade chevron because Windows' native cascade arrow ignores those colors. Consumed by `app._apply_theme`, `rendering.style_axes`, `rendering.draw_session_shading`, `gui/interaction.py` overlays, and `gui/theme_controller._apply_menubar_theme`.
- `CUSTOMIZABLE_THEME_KEYS: list[tuple[str, str]]` â€” curated subset of theme keys exposed to end-user customization in the Settings dialog, as `(key, display_label)` pairs. Currently: `win_bg`, `ax_bg`, `text`, `grid`, `bull_row_bg`, `bear_row_bg`. Non-listed keys always take their base-theme value (acts as the allow-list for `resolve_theme`).
- `DEFAULT_THEMES: dict[str, dict]` â€” mapping `{"light": LIGHT_THEME, "dark": DARK_THEME}` so callers can look up a palette by mode string instead of ternary-ing on `dark_var.get()`.
- `ThemePreset(label: str, mode: str, overrides: dict)` â€” frozen dataclass for one named built-in preset. `mode` is `"light"` or `"dark"` (controls whether `dark_var` flips when applied); `overrides` covers `CUSTOMIZABLE_THEME_KEYS` slots. Used by `gui.theme_editor` to populate the **Built-in presets** strip.
- `PRESET_THEMES: tuple[ThemePreset, ...]` â€” canonical preset registry, in UI display order. Default ship list is `Default Light` / `Default Dark` / `Bloomberg` / `Solarized Light` / `Solarized Dark` / `Nord` / `Dracula` / `Gruvbox Dark` / `Monokai` / `Material Ocean`. The two `Default *` entries are sentinels with empty `overrides` â€” applying them clears the target mode's overrides + flips `dark_var`. Pinned by `tests/unit/gui/test_theme_presets.py` against the keys allow-list, uniqueness, hex-colour shape, and the "every new preset covers all 6 customizable slots" invariant so switching presets is always a complete repaint.
- `resolve_theme(mode: str, overrides: dict | None) -> dict` â€” returns the effective palette for `mode` with user overrides shallow-merged on top. Filters through `CUSTOMIZABLE_THEME_KEYS` so a hand-edited settings file can't inject arbitrary or mistyped keys. Non-string override values are dropped. Used by `ChartApp._apply_theme`.
- `BUILTIN_STARTUP_DEFAULTS: dict` â€” hardcoded fallback values for the user-customizable startup parameters: `{"ticker": "AMD", "compare": "SPY", "interval": "1d", "source": "yfinance", "theme": "light"}`. Source of truth for both the dialog editor and the resolver.
- `STARTUP_DEFAULT_KEYS: list[tuple[str, str]]` â€” drives the dialog row order + labels: `[(ticker, "Default primary ticker"), (compare, "Default compare ticker"), (interval, "Default interval"), (source, "Default data source"), (theme, "Default theme (light/dark)")]`. Adding a new persisted default is a one-tuple insertion here (plus an entry in `BUILTIN_STARTUP_DEFAULTS` and a per-key validation branch in `resolve_startup_defaults`).
- `resolve_startup_defaults(overrides: dict | None, *, intervals=None, sources=None) -> dict` â€” merges sparse `overrides` over `BUILTIN_STARTUP_DEFAULTS` with per-key validation: `ticker`/`compare` are non-empty strings and get `.strip().upper()`; `interval` must be in `intervals` when supplied; `source` must be in `sources` when supplied; `theme` must be `"light"` or `"dark"`. Anything that fails validation falls back to the builtin (silent â€” same pattern as `resolve_theme`). The runtime allow-lists are passed in so this module stays free of imports from `app` / `data` (no cycle).
- `TTK_ROOT_STYLE = "."` â€” sentinel name of the root ttk style that the spec targets first.
- `build_ttk_style_spec(theme: dict) -> list[tuple[str, dict, dict]]` â€” returns a declarative list of `(style_name, configure_kwargs, map_kwargs)` tuples describing every ttk widget class that needs palette (root, `TFrame`, `TLabel`, `TButton`, `Destructive.TButton`, `TCheckbutton`, `TRadiobutton`, `TEntry`, `TCombobox`, `TNotebook`, `TNotebook.Tab`, `Treeview`, `Treeview.Heading`, `TLabelframe`, `TLabelframe.Label`, `TPanedwindow`, `Sash`, `TScrollbar`, `TSpinbox`). The style spec pins clam's normally-inherited chrome slots (`bordercolor`, `lightcolor`, `darkcolor`, `selectbackground`, `selectforeground`, `troughcolor`) to the active palette so buttons, notebooks, label frames, treeviews, and scrollbars do not inherit the light `#eeebe7` / `#cfcdc8` / `#bab5ab` bevels in dark mode. The state maps also cover pressed/active/focus states so hover/press cannot flash back to OS light chrome. Audit `watchlist-entries-full-dark`. The container widgets (`TLabelframe` / `TPanedwindow` / `Sash` / `TScrollbar` / `TSpinbox`) were added in the dark-mode parity sweep (audit `ttk-container-dark`) â€” without them, the Entries and Watchlist tabs (which use `ttk.LabelFrame` + `ttk.PanedWindow` + `ttk.Scrollbar`) fell back to OS-default light-grey chrome under dark theme. `TRadiobutton` was added in audit `radio-hover-dark` for the same reason â€” the per-indicator dialog's "Apply edits to all charts" / "Apply only to <Origin>" radios were inheriting OS-default styling, which paints a light-grey hover halo behind the label in dark mode. The fix mirrors `TCheckbutton`: idle and `active`/`pressed` all pin to `(bg, fg)` so the label background never changes; the radio indicator circle is painted separately by the theme engine and still reacts to selection/press. `Destructive.TButton` is a red-foreground variant reserved for genuinely-destructive actions; idle is red-on-`ax_bg` in both themes, hover / press inverts to white-on-red. Currently consumed by the **PANIC: Flatten All** sandbox button. Audit ``reset-view-style`` (2025) removed the style from the toolbar **Reset View** button because the action is a benign zoom restore, not data loss â€” colouring it red made the reviewer's stock-trader persona anxious about clicking it mid-trade. `map_kwargs` is `{}` for widgets that don't need per-state overrides. Consumed by `app._apply_ttk_style`.
- `ttk_combobox_listbox_options(theme: dict) -> dict` â€” returns option-database keys (`*TCombobox*Listbox.background` etc.) for the readonly Combobox popdown, which is a plain Tk Listbox driven by the option database, not ttk.Style. Consumed by `app._apply_ttk_style`.
- `INTERVAL_PERIODS: dict[str, str]` â€” e.g. `{"1m":"7d","5m":"60d","1h":"730d","1d":"2y","1wk":"10y","1mo":"max"}`. Used by `data/yfinance_source.fetch_live_data` to pick a yfinance `period=` argument that maximizes history while respecting yfinance's per-interval limits.
- `provider_lookback_days(source: str, interval: str) -> int` â€” trailing calendar-day window an on-demand fetch of `(source, interval)` covers. Deep-history vendors (`alpaca` / `polygon`) return generous but **fetch-speed-bounded** per-interval windows (5mâ†’120d, 1hâ†’1460d, 1dâ†’5490d, 1mâ†’20d) â€” they have no yfinance-style 60-day intraday cap, but the whole intraday series is pulled up front on every interval switch / drilldown, so each window is sized to stay ~1 API page (â‰˛10k bars / â‰˛3s) and clear the 5s drilldown deadline (a 2-year 5m window = ~40k bars / ~15s hung the load). Every other source derives from `INTERVAL_PERIODS` (unknown interval â†’ `60d`). **Single source of truth** shared by the Alpaca/Polygon fetchers (default fetch lookback) and `gui/drilldown._day_within_intraday_fetch_window` so fetch depth and drilldown reachability never disagree.
- `GOLDEN_RATIO: float = 1.618â€¦` / `GOLDEN_RATIO_INVERSE: float = 0.618â€¦` â€” canonical Ď† and 1/Ď†. Defining identity `1/Ď† == Ď† - 1`; the major (0.618) and minor (0.382) sections partition any length into the golden split. `CHART_PANE_STARTUP_RATIO` is set to `GOLDEN_RATIO_INVERSE`.
- `CHART_PANE_STARTUP_RATIO: float = GOLDEN_RATIO_INVERSE (~0.618)` â€” fraction of the main window width the chart pane occupies at every launch. The chart claims the golden *major* section (~61.8 %) and the notebook the golden *minor* (~38.2 %) â€” a balanced "unboxing" split (was a flat `0.80`). Consumed by `app.py` during `_build_ui` to pin the `_main_paned` sash via `_apply_forced_sash`. Bypasses `gui/geometry_store.restore_sash` for the main paned â€” the previous design persisted the drag position, which led to the watchlist eating most of the space on subsequent launches if the user accidentally dragged the sash.
- `CHARTSTACK_PANE_STARTUP_WIDTH_PX: int = 220` â€” width of the ChartStack card column in the 3-pane layout, in pixels. Matches `chartstack.card_width_px` so the column comfortably shows one card.
- `compute_main_paned_sashes(main_w: int, *, chartstack_visible: bool, notebook_min_px: int = 280, chart_min_px: int = 200, notebook_width_px: int | None = None) -> list[int]` â€” pure helper returning cumulative `ttk.PanedWindow.sashpos` x-positions for the `_main_paned` layout. Returns `[chart_w]` in 2-pane mode and `[CHARTSTACK_PANE_STARTUP_WIDTH_PX, CHARTSTACK_PANE_STARTUP_WIDTH_PX + chart_w]` in 3-pane mode. **Headline invariant: the notebook (rightmost) pane has the same absolute width in both modes** â€” toggling ChartStack on only steals pixels from the chart, never from the notebook. Notebook width is `max(notebook_min_px, main_w - int(main_w * CHART_PANE_STARTUP_RATIO))` (computed via subtraction to dodge float-precision artefacts). The chart pane has a `chart_min_px` floor so absurdly narrow windows degrade gracefully (chart stays usable; notebook gives up width first). **`notebook_width_px`** (audit `watchlist-width-setting`): a positive int OVERRIDES the golden-ratio notebook width with that absolute pixel width â€” the user's saved watchlist width (`settings["layout.notebook_width_px"]`, captured on File â†’ Save Configuration from the dragged divider). `None` / non-positive / non-int falls back to the ratio; the override is still subject to the `notebook_min_px` + `chart_min_px` floors. Consumed by `app._build_ui` (startup, passing the saved setting) and as the **fallback** path of `app._toggle_chartstack` (when the live boundary can't be measured) and by `app._apply_notebook_width_setting` (Load Configuration).
- `compute_toggle_sashes(main_w: int, notebook_left_x: int, *, chartstack_visible: bool, chartstack_w: int = CHARTSTACK_PANE_STARTUP_WIDTH_PX, chart_min_px: int = 200) -> list[int]` â€” pure helper used by the **ChartStack toggle** path (audit `chartstack-toggle-preserves-notebook`). Unlike `compute_main_paned_sashes` (ratio-based, for startup), this takes the chart|notebook boundary **measured from the live paned before the toggle** and holds it fixed: returns `[chartstack_w, notebook_left_x]` in 3-pane mode and `[notebook_left_x]` in 2-pane mode. The watchlist column does not move; only the chart pane resizes by exactly `chartstack_w`. `main_w` (the **live** `winfo_width`, not the stale `_initial_geometry`) only caps the boundary so the notebook can't be pushed off the right edge; `0` disables the cap. A `chart_min_px` floor nudges the boundary right only in the pathological case where holding it would crush the chart. **Why it exists:** the previous toggle path fed `compute_main_paned_sashes` the stale startup width, so on a resized/maximised window the notebook jumped to ~half the screen.
- `INTRADAY_INTERVALS: frozenset` = `{"1m","2m","5m","15m","30m","1h"}`.
- `is_intraday(interval: str) -> bool` â€” membership test in `INTRADAY_INTERVALS`. Called throughout (app.py, data/normalize.py, streaming/synthetic.py, core/pairing.py, rendering.py indirectly via `draw_session_shading`).
- `classify_session(hour: int, minute: int) -> str` â€” returns `"pre"`, `"regular"`, or `"post"` based on US Eastern wall clock: regular is `[09:30, 16:00)`, post is `[16:00, 20:00)`, everything else is `pre`. Called from `data/normalize.candles_from_dataframe`, `data/synthetic_source._gen_intraday`, `streaming/synthetic.SyntheticStreamSource._make_bar`, `rendering.draw_session_shading`.
- `interval_minutes(interval: str) -> int` â€” converts `"5m"`/`"1h"` to minutes; **raises `ValueError`** on daily+ (callers reaching this on a daily interval are buggy). Used by `data/synthetic_source`, `streaming/synthetic`, `app._schedule_next_bar_fetch`, `app._cache_is_stale`.
- `floor_to_interval(when: datetime, step_min: int) -> datetime` â€” zeros seconds/microseconds and floors `hour*60+minute` to a multiple of `step_min`. Used to line timestamps up with exchange bar boundaries in synthetic data and streaming rollover.

## Dependencies
- Internal: `settings` is imported lazily by `_resolve_initial_palette()` only to read `use_colorblind_palette`; failures fall back to the default palette.
- External: stdlib only (`colorsys`, `dataclasses`, `datetime`). Intentionally zero heavy deps so this module is safe to import from any layer.

## Design notes
- Two theme dicts (not a class hierarchy) â€” palette stays trivially overridable
  and JSON-serializable.
- Declarative ttk spec (`build_ttk_style_spec`) rather than `style.configure`
  calls in `app.py`: light/dark differ only in palette, never in widget
  topology, so structure is expressed once as data. Module stays tk-free.
- Pre/post band colors differ (cool blue pre, warm amber post) so users can
  tell morning vs. evening at a glance.
- `interval_minutes` **raises** on daily+ (silent 1440 would mask buggy
  callers).
- `classify_session` treats overnight as `"pre"` (rare bars, visually
  indistinguishable from pre-market).
- `INTERVAL_PERIODS` picks periods maximizing history within yfinance limits
  (`"1h"â†’"730d"`, `"1m"â†’"7d"`).

## Invariants
- `is_intraday(i)` â†” `i in INTRADAY_INTERVALS`.
- `classify_session(9,30) == "regular"`; `(9,29) == "pre"`; `(16,0) == "post"`;
  `(20,0) == "pre"`.
- `interval_minutes("5m") == 5`; `("1h") == 60`; `("1d")` raises `ValueError`.
- Every key in `LIGHT_THEME` is also in `DARK_THEME` and vice versa.
- `build_ttk_style_spec(theme)` returns the same style names in the same
  order regardless of theme â€” only `configure_kwargs`/`map_kwargs` values
  differ. Guarantees runtime theme-swap can't leave a widget class unstyled.
- `floor_to_interval` expects exchange-local naive datetime (a UTC-aware
  input produces wrong boundaries).
